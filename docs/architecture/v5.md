# Arquitetura v5

## Visão geral

A `v5` mantém os mesmos conceitos da `v4`, mas reorganiza a estrutura para que cada categoria técnica também seja agrupada por domínio.

Em vez de `enums/todo_statuses_enum.rb`, a ideia passa a ser `enums/todo/statuses_enum.rb`. O mesmo vale para serializers, permitted params e value objects. Isso diminui colisão de nomes e melhora a leitura.

## Contexto acumulado até aqui

Até a `v4`, a evolução foi consistente na separação de responsabilidades:

- `v1` começou com a base mínima de `controllers` e `models`
- `v2` extraiu `services` e `serializers`
- `v3` agrupou `services` por domínio
- `v4` acrescentou `enums`, `permitted_params` e `value_objects`

O problema da `v4` não está mais tanto em responsabilidade mal definida. O problema passa a ser navegação. Mesmo com boas abstrações, o código segue espalhado em pastas globais por categoria técnica. Para entender completamente o domínio `todo`, por exemplo, ainda é preciso visitar `services`, `serializers`, `permitted_params` e `enums` separadamente.

A `v5` melhora isso sem abandonar a organização por tipo técnico: cada categoria continua existindo, mas agora também é subdividida por domínio.

## Estrutura

```text
app
├── controllers
│   ├── application_controller.rb
│   ├── todos_controller.rb
│   └── users_controller.rb
├── enums
│   ├── todo
│   │   └── statuses_enum.rb
│   └── user
│       └── roles_enum.rb
├── models
│   ├── application_record.rb
│   ├── todo.rb
│   └── user.rb
├── permitted_params
│   ├── todo
│   │   └── permitted_params.rb
│   └── user
│       └── permitted_params.rb
├── serializers
│   ├── todo
│   │   └── serializer.rb
│   └── user
│       └── serializer.rb
├── services
│   ├── todo
│   │   ├── complete_service.rb
│   │   ├── create_service.rb
│   │   ├── destroy_service.rb
│   │   ├── find_all_service.rb
│   │   ├── find_service.rb
│   │   ├── uncomplete_service.rb
│   │   └── update_service.rb
│   └── user
│       └── create_service.rb
└── value_objects
    └── user
        └── password_value_object.rb
```

## O que melhora em relação à v4

- o domínio aparece na estrutura das pastas, não só no nome do arquivo
- a navegação fica mais consistente entre categorias
- os nomes dos arquivos ficam mais curtos e menos repetitivos

## Leitura arquitetural

A `v5` pode ser lida como uma versão intermediária entre dois mundos:

- ela ainda preserva a visão técnica de topo da aplicação
- mas já injeta forte noção de domínio dentro de cada categoria

Ou seja, ainda não é "tudo de `todo` fica junto", mas já deixou de ser "todo o tipo X fica junto sem contexto".

## Vantagens

- organização mais limpa
- mais espaço para crescer sem gerar arquivos gigantescos por categoria
- reduz redundância nos nomes
- melhora previsibilidade estrutural

## Limitações

- ainda existe separação por categoria técnica no topo de `app`
- para entender tudo sobre `todo`, ainda é preciso visitar várias pastas
- o custo cognitivo de navegação continua razoável em aplicações maiores

## Quando usar

Use a `v5` quando você quer manter categorias técnicas explícitas, mas com uma estrutura mais consistente e escalável por domínio.
